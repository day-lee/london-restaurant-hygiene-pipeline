## Ingestion: FSA 다운로드해서 S3 버킷에 올리기 

- docker-compose 환경에서는 로컬 venv에 pip install을 하지 않아도 된다. 컨테이너 내부에 기본적으로 설치가 되어있음
- 처음에는 단일 보로우로 프로토 타입을 검증한 뒤에 33개의 보로우를 병렬 처리하도록 확장함 

### Dag & Task
- 프로젝트 적재 기준점을 임의의 과거 시점으로 '2026, 7, 1' 설정하고, catchup=False를 켜놔서 1일 이후지만 밀린 데이터는 건너뛰고 오늘 수동으로 누른 배치만 실행되도록 함 
- `_extract_xml_to_s3_streaming()`파이썬 함수를 하단에서 Task로 만들려면 PythonOperator()를 사용해 `extract_xml_to_s3`라고 포장해서 사용한다. 

### 전체 구조 
- requests를 사용해 메모리 버퍼에 올려서 스트림 형태로 XML 파일을 받아온다.
- s3가 인식할 수 있는 파일 형태로 위장하기 위해 BytesIO를 사용한다. 
- s3 hook으로 스트리밍 전송해준다. 
- task를 만들어서 호출한다. 

### BytesIO가 하는 일 
- `memory_buffer = BytesIO()` 
-  Bytes(바이너리 데이터)를 I/O(In/Out, 읽고 쓰기) 하는 메모리 공간
- .write(), seek(), close() 메모리에서 쓰고, 메모리 버퍼 처음 부분으로 커서 돌려놓고, 해제해주는 메소드 
- 하드디스크에 파일 만들지 않고, 램 메모리 영역에 임시로 파일인척 위장해서 가상의 파이썬 파일을 s3로 보냄. 
- request의 stream 인자는 데이터 다운로드시 한번에 메모리에 올리지 않고 쪼개서, 스트림 형태로 받아온다는 의미 
- 만약 이 처리를 하지 않으면 에어플로우 컨테이너 메모리가 가득 차서 out of memory 에러 발생 
 `for chunk in response.iter_content` 파이썬 제너레이터 문법 
- chunk_size=8192는 약 8kb 단위로 메모리 버퍼에 옮김. 
- 33개 구 4mb기준으로 스케일 아웃해서 128kb로 확장. 구당 For 루프 횟수가 약 500회에서 30회 내외(128kb * 33개 = 4.1MB)로 로 대폭 감소하여 CPU 및 네트워크 I/O 오버헤드 최적화

### api 요청 vs XML 파일 다운로드 
- 데이터의 시간 범위, 수집 목적에 따라 선택

- 현재 프로젝트는 주 단위 배치 작업이고, 데이터 웨어하우스, 레이크 구축이 목적이라서 XML 덤프 다운로드가 적합함
- api request limit 걱정이 없고 안정적으로 수집 가능 

- API 요청은 실시간성, 분, 초단위의 고주기의 짧은 시간 간격 단위일 때 
- 필요한 만큼 쪼개서 트래픽을 최적화 시킬 수 있다. pagination, filter 사용 가능

### 중간 구조 변경
- Ingestion 단계에서 1.FSA ➔ S3 XML 다운로드로 로우 데이터를 수집하고, 수집이 끝나면 2.S3 ➔ DB 적재를 분리해서 설계했었음
- 시니어의 리뷰를 받고 두 과정을 메모리 버퍼에 올린 상태에서 동시에 진행하기로함.  
- s3를 중간 저장소로 이용할 필요가 없음  
- 인그레스 진행 중에 메모리에 데이터가 이미 올라와 있으니, 이 시점에 DB 적재까지 한번에 처리해라. 
- 이그레스 비용: 보통 클라우드 환경에서 데이터를 넣는 것(인그레스)은 무료이나, 이그레스는 비용을 청구함. 
- 이전 설계에서는 데이터를 넣어두고나서, 다시 이그레스해서 빼내야 했음. 
- FinOps 관점에서 이 구조는 이그레스 비용이 너무 많이 발생함.  

- Ingress: 외부에서 시스템 내부로 데이터가 들어오는 흐름 (Inflow)
- Egress: 시스템 내부에서 외부로 데이터가 나가는 흐름 (Outflow)

### 데이터 레이크(원본 데이터 보존) vs 데이터 웨어하우스, 데이터 마트 (분석, 서비스 영역) 
- 시니어가 왜 S3 스텝이 필요한지 물어봤음 

#### 데이터 레이크 DL
- s3는 Single Source of Truth로 사용됨. 
    - s3 glacier는 아카이브 등급으로 과거 데이터 저장가능. 비용이 굉장히 적음 
- 로우 데이터 백업: 외부 공공 데이터의 특성상 변경되거나 제공 중단 될 수 있음 
- 감사 오딧용(data lineage)
- 외부 데이터 접근 불가능: 네트워크 문제 등, pipeline re-run
- data replication, database redundancy 미래 유실 대비 등.. 

#### 데이터 웨어하우스 DW
- 회사 전체 데이터 대상 
- 비즈니스에 필요한 핵심 필드만 가공해서 postgreSQL에 upsert함 
- 데이터 웨어하우스 DW 전문 솔루션: 대용량 분석에 특화된 AWS Redshift, Google BigQuery, SnowFlake 

#### 데이터 마트 DM
- 웨어하우스에서 특정 부서나 목적에 맞게 골라내서 만든 저장소 
- 작고 단순하게 구성되어 빠른 쿼리 속도로 현업 담당자들이 바로 쓸수 있게 가공되어 있음
