## Docker compose로 Airflow 2 & PostgreSQL 16를 띄우기 
- Airflow 3이 출시되었지만 실무에서는 아직 2를 많이 쓰고 있을 것으로 예상함.
- airflow는 메타데이터를 저장할 DB가 필요하다. 로컬에서는 기본적으로 SQLite를 사용한다. 실무에서는 병렬처리를 위해 PostgreSQL을 사용한다. 
- PostgreSQL 컨테이너에서 메타데이터용 DB와 분석용 데이터 웨어하우스 DB를 물리적으로 분리한다.   

### Container
- 2개의 DB 컨테이너: airflow_postgres_meta, analytics_postgres_dw
- 3개의 Airflow 컨테이너: airflow_init, airflow_webserver, airflow-scheduler

### Data Management (Volume)
#### 1. Bind Mount
- 컨테이너가 띄워진 상태에서 코드를 수정하면 바로 반영될 수 있음 
e.g.,
```
volumes:
  - ./dags:/opt/airflow/dags
```
- 내 컴퓨터의 ./dags 폴더와 컨테이너 안의 /opt/airflow/dags 폴더를 실시간으로 동기화(연결)해라

#### 2. Named Volume 
- `postgres_meta_data:/var/lib/postgresql/data`
- 도커가 관리하는 안전한 가상 저장 공간에 데이터를 저장해라 
- `./, /` 경로로 시작하지 않고 영어 단어로만 시작한다. 
- 이 가상 공간은 내 컴퓨터 내부의 직접 접근 할 수 없는 도커 전용 비밀 격리 구역임. VM내부 하드디스크 
- 로컬 폴더랑 동기화하면 권한(permission), 성능 저하 문제가 생길 수 있어서 컨테이너를 끄고 켜도 데이터가 날아가지 않도록 보호하는 목적. 
- docker compose down으로 컨테이너를 완전히 삭제해도, 이 Named Volume 영역은 사용자가 명령어(-v 옵션)로 직접 지우지 않는 한 절대 지워지지 않고 영구 보존

#### 3. airflow 전역 공유 환경 변수 
- x-airflow-common: &airflow-common는 모든 airflow 서비스(웹서버, 스케쥴러, 이닛) 이 공유하는 방이다. 
- 내 컴퓨터와 도커 환경을 일치시키는 `PYTHONPATH: /opt/airflow`를 추가하여 앞으로 커스텀 폴더를 새로 만들어도 범용적으로 자유롭게 사용가능.
- volume에도 추가해줘야함.

### docker-compose.yml 작성 후 `docker compose up -d`로 올리기 
- .env로 민감 정보 옮기기 
- port 연결 확인 
- volume 경로 확인 

### 검증 
1. `docker ps`로 status 확인 
2. http://localhost:8080 접속 확인 
  - 올린 뒤 변경 사항 있다면 `docker compose up -d --build`로 바로 반영가능하지만 
  - 환경 변수(로그인 디테일) 경우 `docker compose down -v`해서 볼륨까지 지운 뒤 다시 생성하는게 깔끔함 
3. Airflow 스케쥴러 상태 검증
  - UI에  "The scheduler does not appear to be running..." 와 같은 경고 문구가 없다. 
4. 분석용 DW DB 외부 접속 테스트 
  - DBeaver에서 연결 설정
